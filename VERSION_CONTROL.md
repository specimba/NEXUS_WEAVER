# NEXUS Visual Weaver — Version Control & Recovery Guide

Professional Git workflow with GitHub for the NEXUS Weaver project. This system
protects against sandbox resets and provides clean rollback points.

## Quick Reference

| Action | Command |
|---|---|
| **View available versions** | `git tag -l` |
| **Rollback to a version** | `git checkout v5.25` |
| **Restore .env after wipe** | `./scripts/restore-env.sh` |
| **Push current .env to GitHub** | `./scripts/push-env-to-github.sh` |
| **Create a new version tag** | `git tag -a v5.26 -m "description" && git push origin v5.26` |
| **View commit history** | `git log --oneline -20` |

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  GitHub Repo (specimba/NEXUS_WEAVER)                        │
│  ├── Source code (163 files, lean)                          │
│  ├── Tags: v5.25, v5.26, ... (rollback points)             │
│  ├── GPG-signed commits                                     │
│  └── Encrypted Secrets:                                     │
│      ├── MODAL_TOKEN_ID / MODAL_TOKEN_SECRET                │
│      ├── MODAL_PROXY_KEY / MODAL_PROXY_SECRET               │
│      ├── HF_TOKEN                                           │
│      ├── BROWSERLESS_TOKEN                                  │
│      └── NEXUS_ENV_BLOB (full .env base64-encoded)          │
└─────────────────────────────────────────────────────────────┘
                          │
                    git clone / pull
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  Sandbox (volatile — may reset at any time)                 │
│  ├── Source code (from git)                                 │
│  ├── .env (restored from GitHub Secrets)                   │
│  ├── db/custom.db (regenerated by prisma db:push)          │
│  ├── node_modules (regenerated by bun install)              │
│  └── public/gallery/ (regenerated by pipeline runs)         │
└─────────────────────────────────────────────────────────────┘
```

**Key principle:** Token values NEVER appear in git history. They live only as
encrypted GitHub Secrets. The source code reads from `process.env`, and a
recovery script restores `.env` from GitHub Secrets after a wipe.

## Recovery After Sandbox Wipe

### Option A: Full restore via GitHub Actions workflow (recommended)

```bash
# 1. Clone the repo
git clone https://github.com/specimba/NEXUS_WEAVER.git
cd NEXUS_WEAVER

# 2. Install dependencies
bun install

# 3. Set your GitHub PAT
export GH_PAT="<your-github-pat-here>"

# 4. Run the restore workflow
gh workflow run restore-env.yml

# 5. Wait for it to complete, then download the artifact
gh run list --workflow=restore-env.yml --limit=1
# Note the run ID, then:
gh run download <RUN_ID> -n env-file
mv env-file/restored.env .env

# 6. Create the database + start dev server
bun run db:push
bun run dev
```

### Option B: Quick restore via script

```bash
git clone https://github.com/specimba/NEXUS_WEAVER.git
cd NEXUS_WEAVER
bun install
export GH_PAT="<YOUR_GITHUB_PAT>"
./scripts/restore-env.sh    # writes .env template (fill tokens manually)
bun run db:push
bun run dev
```

## Creating Version Tags (Rollback Points)

Before any major change, create a tag so you can roll back:

```bash
# Tag the current state
git tag -a v5.26 -m "v5.26 — description of what changed"

# Push the tag to GitHub
git push origin v5.26
```

### Rollback to a previous version

```bash
# View available versions
git tag -l

# Roll back (detached HEAD — safe, read-only)
git checkout v5.25

# Or roll back AND make it the new main (destructive)
git reset --hard v5.25
git push --force origin main
```

## Pushing .env Updates to GitHub Secrets

When you change a token (e.g., rotate the Modal PAT), push the new .env:

```bash
export GH_PAT="<YOUR_GITHUB_PAT>"
./scripts/push-env-to-github.sh
```

This stores:
- Each token as an individual GitHub Secret
- The full .env as `NEXUS_ENV_BLOB` (base64-encoded)

## Commit Workflow

All commits are GPG-signed (key: `21897A2CC2FA8793`). The signing key is
configured in `.git/config`:

```ini
[user]
    signingkey = 21897A2CC2FA8793
[commit]
    gpgsign = true
[tag]
    gpgsign = true
```

### Commit message convention

```
<type>: <short description>

<longer explanation if needed>

Files changed:
- path/to/file (what changed)
```

Types: `feat`, `fix`, `security`, `chore`, `docs`, `refactor`, `merge`

## What's Tracked vs. Ignored

### Tracked (in GitHub repo)
- `src/` — all source code
- `modal-apps/` — Modal deployment scripts
- `prisma/schema.prisma` — database schema
- `scripts/` — recovery + deployment scripts
- `.github/workflows/` — CI/CD workflows
- `src/lib/secrets.ts` — reads from env, NO hardcoded tokens
- `.env.example` — template (no real values)

### Ignored (regenerated)
- `node_modules/` — `bun install`
- `db/custom.db` — `bun run db:push`
- `.env` — `scripts/restore-env.sh`
- `public/gallery/*` — pipeline runs
- `upload/` — user reference files
- `dev.log`, `*.log` — runtime logs
- `.next/` — build cache

## Security Notes

1. **Token values never in git history.** GitHub secret scanning passes.
2. **GPG-signed commits.** Verified badges on GitHub.
3. **PAT has limited scope.** Expires 2026-10-05. Rotate if compromised.
4. **`.env` is gitignored.** Even if accidentally committed, it's excluded.
5. **`secrets.ts` reads from `process.env` only.** No hardcoded fallback values.

If a token is compromised:
1. Rotate it at the provider (Modal, HuggingFace, Browserless)
2. Update `.env` locally
3. Run `./scripts/push-env-to-github.sh` to push the new value
4. Re-deploy any Modal apps that use the old token
